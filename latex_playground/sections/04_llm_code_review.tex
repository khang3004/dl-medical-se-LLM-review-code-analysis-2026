\section{LLMs as Code Review Agents: Evaluation and Alignment}
\label{sec:llm_review}

\paperCard{LLMs as Code Review Agents: A Rapid Review and Experimental Evaluation with Human Expert Judges}{M. Kawalerowicz, M. Pietranik, K. St\k{e}pniak}{ICCCI 2025}

\subsection{Problem Statement \& LLM-as-a-Judge}
Quality assurance through code review is a vital phase of the software development lifecycle (SDLC), but it is resource-intensive. Automated tools like static linters are limited to syntactic rules, failing to assess semantic properties like readability, SOLID design principles, or security risks. 

Recently, Large Language Models (LLMs) have been proposed as automated code review agents under the \highlight{LLM-as-a-Judge} paradigm. While LLMs are cheap and fast, assessing their alignment with human expert consensus and understanding their biases is crucial before deployment in production pipelines.

\subsubsection{Experimental Setup}
\begin{itemize}
    \item \textbf{Dataset:} 75 source code files in Java, C\#, Python, JavaScript, and TypeScript, selected to represent varying code quality.
    \item \textbf{Evaluators:} 6 LLMs (Claude 3.5 Sonnet, Claude 3.5 Haiku, GPT-4o-mini, Mistral Large, Ministral-8b, Falcon3-7B) compared against 3 human software developers (5–15 years of industry experience).
    \item \textbf{Metrics:} Code is scored on a scale of 1–10 across 12 quality metrics: comments ($m_1$), DIP ($m_2$), formatting ($m_3$), ISP ($m_4$), KISS ($m_5$), LSP ($m_6$), naming ($m_7$), OCP ($m_8$), performance ($m_9$), DRY/repetition ($m_{10}$), safety ($m_{11}$), and SRP ($m_{12}$).
\end{itemize}

\subsection{Statistical Rigor \& Consensus Pipeline}
The study establishes a pipeline to validate the statistical significance of their findings:
\begin{enumerate}
    \item \textbf{Normality Check:} The Shapiro-Wilk test is performed on all score distributions. The resulting $p$-values are less than $0.05$ ($p < \alpha$), indicating that the scores do not follow a normal distribution.
    \item \textbf{Correlation Metric:} Because the data is non-normal, the non-parametric \highlight{Spearman Rank Correlation} ($\rho$) is used to measure monotonic relationships:
    \begin{equation}
        \rho = 1 - \frac{6 \sum d_i^2}{n(n^2 - 1)}
    \end{equation}
    where $d_i$ is the difference between ranks.
    \item \textbf{Ground Truth Modeling:} Human developer consensus is modeled to establish a baseline:
    \begin{equation}
        GT(f_i) = \text{Median}(S_{Human1}(f_i), S_{Human2}(f_i), S_{Human3}(f_i))
    \end{equation}
\end{enumerate}

\subsection{Advanced MLE/AIE Proposed Extensions}
The evaluated LLM review configuration is limited by its single-file context and lack of runtime feedback. We propose three system-level extensions.

\subsubsection{1. Project-Level Dependency Graph-RAG}
Code files cannot be reviewed accurately in isolation. We propose building a multi-file dependency graph $G = (V, E)$ to represent the code repository, where $V$ represents code entities (methods, classes, modules, database schemas) and $E$ represents relationships (inheritance, method calls, imports, data flows) extracted via Abstract Syntax Tree (AST) parsing.
During code review of file $f$, the system queries the graph database using Graph-RAG \cite{graph_rag_2024}:
\begin{equation}
    C_{context} = \text{Retrieve}(f, G)
\end{equation}
This context (containing class definitions and dependent APIs) is injected into the prompt. This allows the LLM to verify project-wide API contracts and detect LSP/DIP violations across file boundaries.

\subsubsection{2. Execution-Guided Dynamic Review Loop}
LLM code evaluations can sometimes generate hallucinations. We propose integrating a sandboxed runtime compiler-debugger loop:
\begin{enumerate}
    \item The LLM suggests structural modifications to resolve a bug or design violation.
    \item The proposed code is compiled inside a secure Docker container, which runs static checks (Semgrep/SonarQube) and dynamic tests (unit tests, execution profilers).
    \item The execution feedback (stdout, stderr, compiler logs) is parsed and returned to the LLM agent:
    \begin{equation}
        \text{Prompt}_{new} = \text{Prompt}_{old} \cup \{ \text{Compiler Logs} \}
    \end{equation}
    \item The LLM iterates on the code until the tests pass. This ensures that the generated code is both architecturally sound and functionally correct.
\end{enumerate}

\subsubsection{3. Multi-Agent Consensus Debate Protocol}
Instead of relying on a single LLM, we propose a multi-agent panel representing specialized personas: a Security Architect, a Performance Engineer, and a Clean Code Reviewer.
Let $s_a^0$ be the initial score vector from agent $a$. A Moderator Agent computes the variance of the scores:
\begin{equation}
    \sigma_j^2 = \text{Var}(s_{1,j}^0, s_{2,j}^0, s_{3,j}^0) \quad \text{for metric } j
\end{equation}
If $\sigma_j^2 > \tau$ (threshold), the Moderator triggers a multi-turn conversation. The agents debate their scores, referencing specific line numbers. This debate continues until the variance drops below the threshold, yielding a more stable score vector.
